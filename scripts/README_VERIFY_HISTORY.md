# 验证脚本历史口径说明

## 目的

记录仓库内历史验证脚本、内部脚本与正式验收口径之间的边界，避免误用。

## 当前分类

### 1. 正式验收入口

- 本轮收口范围内，**未新增**任何合成主循环正式验收入口。
- 尤其是：`archive:verify:merge-loop` **不是**正式验收入口。

### 2. archive/internal 脚本

以下口径统一归入历史/内部/归档用途：

- `archive:verify:merge-loop`
- 任何用于局部模块排查、历史回归、临时编译运行的脚本
- 尚未被 README / MAIN_LOOP_VERIFICATION / 路线图共同确认的验证命令

这些脚本只能用于：

- 历史留档
- 内部排错
- 开发期局部回归
- 人工辅助理解模块行为

不得用于：

- 宣称“合成循环已具备正式验收入口”
- 作为对外唯一验收凭据
- 替代未来统一主循环验收链路

## 与当前实现的对应关系

- 当前仓库内已有合成系统相关代码：`assets/scripts/merge/`、部分 UI 接线、增益逻辑。
- 但这些实现仍处于“代码存在、边界已说明”的状态。
- `archive:verify:merge-loop` 即便在历史版本中可运行，也只代表内部验证素材存在，不代表正式验收体系已经建立。

## 维护要求

后续若新增正式验收命令，必须同步更新：

1. `README.md`
2. `MAIN_LOOP_VERIFICATION.md`
3. `roadmap.md`
4. `package.json`
5. 本文档

在这五处口径未同步前，一律按 archive/internal 处理。


## 历史失败文档的处理原则

- 历史失败结论只代表对应时间点、对应分支或对应提交快照。
- 当前仓库若未提供新的正式主循环验收命令，则统一结论应写为：**暂无正式主循环验收入口**，而不是同时写成“正式验收已通过”或“正式验收失败中”。
- 因此，旧失败文档可以保留为历史记录，但不得再与当前状态并列表述为现行结论。
- 一切当前口径以 `README.md`、`roadmap.md`、`MAIN_LOOP_VERIFICATION.md`、`package.json` 与本文同步后的内容为准。
