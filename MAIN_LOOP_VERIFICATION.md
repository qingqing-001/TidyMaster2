# 交付仓库中的验证入口边界说明

## 文档目的

本文件用于按交付仓库口径说明：哪些内容属于当前对外可引用的正式入口，哪些内容仅属于历史留档、内部脚本或局部验证材料。

## 当前正式结论

- 当前仓库没有对外交付的正式 `main loop` / `merge loop` 验收命令。
- 当前稳定、仓库级、可直接执行的正式入口包括：`npm run type-check`。
- `npm run verify:chapter1-drag-loop` 当前保留为第 1 章拖拽闭环的**正式命名导航入口**：它在 `package.json` 中只输出统一说明，用于引导审查者理解当前实现边界与历史背景。
- `archive:verify:merge-loop` 仅为 **archive/internal** 历史占位脚本，不执行正式合成交付验收。

## 如何理解“交付可核对”与“历史可追溯”

当前仓库可以被交付和核对的，是：

- 代码与资源已入库这一事实
- `npm run type-check` 这一稳定静态入口
- 顶层文档对正式入口、历史资产、命名边界的统一说明

当前仓库可以被追溯、但**不能**直接升格为正式交付结论的，是：

- `archive/history/root-signals/` 下的历史验证快照
- `archive/history/internal-scripts/` 下的历史 `verify-*` 脚本
- 历史 report / note / setup / verification 材料
- 代码中已存在但尚未形成统一正式仓库级验收链路的模块实现

## 与当前实现范围的对应关系

从代码结构与资源布局看，仓库已经交付了较多实现内容，例如：

- 场景与资源资产
- 拖拽、折叠、合成、UI、音频等相关模块代码
- 微信小游戏构建侧配置

但这些“已有代码/已有资源”并不自动等于：

- 已提供覆盖全项目的正式验收命令
- 已形成可一键复核的 main loop / merge loop 交付链路
- 可直接对外宣称“整体功能已正式验收完成”

因此，当前更准确的交付表述是：**仓库已交付项目源码、资源与文档边界说明，但尚未交付覆盖全项目的正式主循环/合成循环验收入口。**

## 对外引用要求

若在交付说明、评审记录或后续文档中提到验证入口，必须遵守以下要求：

- `npm run type-check` 可作为当前正式入口引用。
- `npm run verify:chapter1-drag-loop` 必须标注为“正式命名导航入口”。
- `archive:verify:merge-loop` 必须标注为 `archive/internal` 历史占位脚本。
- 历史失败、历史通过、历史验证报告均必须说明其仅代表对应时间点或对应快照。
- 不得把 archive/history 中的材料与当前正式交付入口并列描述为同等级结论。

## 备注

本文件用于统一交付口径，不代表新增了新的正式验收能力，也不代表补建了一键主循环验证链路。
